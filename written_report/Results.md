We managed to complete every project goal laid out at the beginning of this project. Firstly, we successfully read in and parsed the flight routes data to construct a Graph object where vertices represent Airports and edges represent connecting flight(s); weights were set such that a lower weight represents higher throughput between Airports. The Graph is stored internally as an n*n adjacency matrix, where n is the number of unique Airports.

Our Graph object can be used as an argument to create the BFS class, which uses the AirportTraversal::Iterator to iterate through the complete dataset in Breadth-First order. Beginning at the first Airport “AER”, the BFS traverses through the dataset in the order shown in appendix 1.

We also implemented the Dijkstra algorithm as a member function of Graph. Graph::Dijkstra accepts a starting and ending Airport as arguments (either by name or by id) and returns a vector of Airports representing the path with the lowest total weight (highest throughput). For instance, the lowest-weighted path between AER (Sochi) and LAX (Los Angeles) is **AER(Sochi), DME(Moscow), BKK(Bangkok), ICN(South Korea), LAX(Los Angeles)**. If a valid path does not exist between two airports, an empty vector is returned.

Additionally, we used the adjacency matrix to run a Page Rank analysis. This is achieved by creating a Markov Chain matrix and then finding its corresponding steady-state vector, representing the relative Page Rank score of an airport. Matrix operations were carried out using the Eigen library. Weights were assigned in the Markov Chain matrix such that higher weights represented a higher probability of travel from one Airport to another, and a dampening factor was applied to all weights to simulate random behavior. The top 10 Airports with the highest Page Rank scores were **ATL (Atlanta), ORD (O’Hare), LAX (Los Angeles), DFW (Dallas), CDG (Paris), LHR (London), SIN (Singapore), PEK (Beijing), DEN (Denver) and FRA (Frankfurt).** We found this to be a successful analysis, as it was closely correlated with the 2017 list of busiest airports by aircraft movements and, to a lesser extent, the 2020 list of busiest airports by passenger traffic. 

Finally, we attempted to answer our leading question - To what extent can Dijkstra’s Algorithm be used for ranking? We predicted that the probability of an airport appearing in a random lowest-weighted path is positively correlated with the Airport’s “importance”. To test this, we sampled 1000 pairs of randomly selected Airports A and B and counted the number of times an airport C was part of the lowest-weighted path from A to B. We noted a disadvantage to this method: there are 3,425 Airports in the dataset, yielding a total of 11,730,625 permutations of start/ends. This means that our sample size of 1000 is relatively imprecise. However, unlike Page Rank, this method can be scaled easily by adjusting the sample size for a tradeoff between faster computation and accuracy. The top 10 Airports based on our Dijkstra estimation were **LAX (Los Angeles), LHR (London), ATL (Atlanta), SIN (singapore), ORD (O'Hare), JFK (New York), ANC (Alaska), MIA (Miami), SEA (Seattle) and BKK (Bangkok)**. 

As a stretch goal, we implemented Spearman’s correlation formula. Spearman’s correlation yields a value between -1 and 1 for the amount of correlation between two equally-sized rankings, with 1 representing perfect correlation. The Page Rank ranking and the Dijkstra’s estimation ranking had a Spearman correlation coefficient of **0.4225**. On the same system, the Page Rank analysis took roughly 10 seconds to complete while the Dijkstra estimation took approximately 300 seconds. Our conclusion, therefore, is that Dijkstra’s algorithm can be used to approximate a ranking for a set of Nodes in a graph. However, Page Rank is a much more accurate and efficient approach, given that matrix operations can be carried out efficiently.
